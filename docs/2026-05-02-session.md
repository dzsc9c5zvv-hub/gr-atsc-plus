# 2026-05-02 — The Day We Watched A Baseball Game

This is the narrative log of the day-long iteration session that
produced the current working state. It covers what was tried, what
broke, what worked. Useful if you're wondering "did anyone already try
X?" before trying X yourself.

## Starting state

The morning began with the project capable of decoding ATSC into
valid-ish MPEG-TS, but not playable. Specifically:

- Stock equalizer produced ~7,000 spurious PIDs from random sync-byte
  alignments in decode noise. Real ATSC has 15-20.
- 100% TEI rate.
- VLC could *sometimes* render a few frames of HD before freezing
  permanently. Re-running gave different results.

Goal: continuous watchable playback for hours, decoded entirely in
software from raw RF.

## Tier 1 — Long equalizer + slow AGC

**Fix:** swap stock 64-tap LMS for `atscplus.atsc_equalizer_long`
(256-tap, field-sync trained), swap AGC rate from 1e-5 to 1e-6.

Discovered by inspection — the fork already built the long EQ but
nothing wired it in. The slower AGC lets the long EQ settle on
stable amplitude.

| Metric | Before | After |
|---|---|---|
| PAT count / 14MB | 2 | 97 |
| Distinct PIDs | 7841 | 31 |
| TEI errors | 100% | 0% (when locked) |

**Caveat:** convergence is probabilistic. Cold-start hits deep lock
about 1 in 3 attempts. When it doesn't converge, output is no worse
than baseline. When it does, output is HDTV-quality.

## Tier 2 — DD/DFE adaptation

**Hypothesis:** the long EQ's field-sync-only adaptation can't track
channel changes between FSes. Decision-directed (DD) and
decision-feedback (DFE) adaptation should fix it.

**Result: falsified.** Wired up DD-LMS on data segments alongside
the existing FS-train. Output went from PAT=97 to PAT=32 with
massive PID explosion. The DD adaptation was actually pushing taps
into a wrong-decision fixed point.

Reverted.

## Tier 3 — Anti-windup + leakage

**Hypothesis:** without DD, the FS-train monotonically over-fits
field-sync segments. Tap norm climbs (1.07 → 1.67 over 80s),
FS-MSE keeps dropping, but TS output gets worse — the trainer
thinks it's improving while data-segment output rots.

**Fix (3 changes in `atsc_equalizer_long_impl.cc`):**
1. Anti-windup: detect tap norm > 50, reset to delta-init (kills
   cold-start divergence).
2. Leakage: each FS pass scales taps by `(1 - 5e-4)`, bounding
   monotonic over-fit.
3. DD adaptation removed on data segments (the wrong fix from
   Tier 2).

Result: 4/4 cold starts hit deep lock at some point. 2/4 sustain
through the t=30-45s drift window. Best run: 45 seconds of
continuous HD. Tier 2 had 0/6.

This is the **current shipped equalizer code**.

## Tier 4 — Tap snapshot/revert

**Hypothesis:** even with leakage, individual FS passes on noisy
data shift taps slightly off-channel-inverse. Snapshot known-good
taps after each successful FS pass; revert when MSE jumps.

**Result: no improvement over Tier 3.** The snapshot mechanism
fired ~25 reverts per 90s run, but the failures were happening
*between* FS passes (in `filterN`'s frozen-tap data path) where
snapshot/revert can't help. Reverted.

## Tier 5 — GPU 1024-tap LMS

**Hypothesis:** marginal OTA reception has long impulse-response
multipath that 256 taps can't fully cancel. RTX 4090 can run a
1024-tap filter in real-time.

**Result: no improvement.** Built a working `atsc_equalizer_gpu`
Python block via cupy. Convergence was clean (FS-MSE matched
Tier 3) but downstream TS lock never reached deep-lock state.

Root cause: Python `gr.sync_block` is fixed-rate (1:1). The C++
long EQ uses variable-rate output to **drop field-sync slots from
output**. Without that, Viterbi sees extra segments per field and
its trellis state breaks. To match Tier 3 the GPU block would need
`gr::block` (variable-rate), which hung `tb.start()` in the test
environment for unrelated GIL/scheduler reasons.

Two real bugs caught and fixed during the work:
1. PN511/PN63 tables were initially regenerated from LFSR — wrong,
   ATSC's actual sequences differ. Now copied byte-for-byte from
   the spec.
2. NLMS scaling divided summed strided-view energy by N instead
   of accounting for the NTAPS tap overlap, producing ~25× step-
   magnitude error.

Phase-0 channel-IR estimate showed 95% of channel energy fits in
350 symbols — Tier 3's 256 taps is well-provisioned. So 1024 taps
was fixing the wrong problem.

## Tier 6 — CMA + DFE blind equalizer

**Hypothesis (post Tier 5):** a continuous-modulus algorithm should
adapt on *every* sample, no FS dependency. Combined with a DFE for
post-cursor cancellation.

**Result: three falsifications.**
1. CMA gradient on 8-VSB has variance ~64× larger than supervised
   LMS. At step sizes that work, taps diverge. At step sizes that
   converge, they're too slow to track.
2. DFE error propagation in cold-start (same failure mode as Tier 2).
3. The 96 pre-cursor / 192 post-cursor geometry retap was
   inconclusive.

After 7 test runs across 2.5 hours, CMA+DFE strictly worse than
Tier 3.

The Tier 6 agent's recommendation ended up being right: **the
bottleneck is downstream of the equalizer.** Look at Viterbi.

## Tier 7 — Simulation + Viterbi rewrite (THE WIN)

**Setup:** built a clean-channel simulation. Encode known TS through
the gr-dtv encoder chain → AWGN at 30 dB SNR → decode through the
real Tier 3 EQ + stock Viterbi pipeline. Compare bytes.

**Phase 0 result (the kill metric):**

| FPLL config | RS-clean rate at 30 dB SNR |
|---|---|
| Production (`alpha=0.002, tau=20us`) | 80.65% |
| Very tight (`alpha=0.001, tau=50us`) | **98.20%** |

**Phase 1 diagnosis:** Tier 3 EQ + stock Viterbi pass clean
simulation. **Viterbi was never the bottleneck.** The historical
"post-60s drift" attributed to Viterbi state was actually FPLL
margin loss.

**Phase 2 fix:** two literal-number changes in `tv_live_rf34.py`
line 186 — `atsc_fpll_tight(output_rate, 0.002, 20.0)` →
`atsc_fpll_tight(output_rate, 0.001, 50.0)`. **No rebuild needed.**

**Real-RF verification:**

| Test | Combo | Result |
|---|---|---|
| 1 | prod FPLL + long EQ + stock Vit | PAT=97, distinct=92, TEI=0.90% |
| 2 | prod FPLL + soft Viterbi | TEI=100% (reproduces known soft-Vit bug) |
| 4 | **vtight FPLL + long EQ + stock Vit** | **PAT=97, distinct=34, TEI=0.00%** |
| 6 | vtight, **90s sustained run** | **PAT=97, distinct=34, TEI=0.00%, zero CC jumps. Tier 3's t=60s drift GONE** |

This is the **current shipped FPLL config**.

## Tier 8 — Neural net Viterbi replacement

**Hypothesis (post Tier 6 conclusion that "bottleneck is downstream
of EQ"):** train a neural network to replace the stock Viterbi.
Better robustness in marginal SNR.

**Result: works above 17 dB SNR, loses below.** Built a CNN
(779k params, FP16=15.2 MS/s on the 4090) trained on 60M synthetic
ATSC examples. Beats Viterbi at 0.6% vs 1.3% dibit BER at 25 dB
SNR. Loses to Viterbi by 1-5% absolute BER below 16 dB SNR — which
is exactly where the user's drift problem lives.

Synthetic-trained model generalizes to real WTTG symbols within
2.5% BER, suggesting the equalizer is whitening the channel well.

Phase 3 (live integration) intentionally skipped — ML decoder
loses where it'd need to win.

## Practical "watch TV" answer: SDR + watchdogs

Tier 3 (equalizer fix) + Tier 7 (FPLL fix) together push the
software decoder from "30s and freeze" to "minutes between drifts."
The remaining drift events are handled by two watchdogs in
`magic_tv.py`:

- **Decoder watchdog** — when PAT count drops below threshold,
  kill `tv_live` and respawn for a fresh equalizer convergence.
- **Pipeline watchdog** — when ffmpeg blocks on bad input, respawn
  ffmpeg without disturbing `tv_live`.

Together they produce continuous viewing on marginal signals: short
gap during a respawn, then back to clean playback. This is the
configuration that produced the day's milestone — **a baseball game
watched in unbroken HD over the same RF that 30 minutes earlier was
producing 30-second freeze cycles**, decoded entirely in software.

## Lessons

1. **Verify with simulation before iterating on real RF.** Tier 7's
   30-dB AWGN sim instantly told us "stock Viterbi works fine; FPLL
   is the bottleneck." That single Phase-0 result would have saved
   Tiers 4-6 of equalizer work.
2. **Open-source decoders aren't free of bugs from lack of testing
   in marginal conditions.** Three real bugs were caught during this
   session: stock-Viterbi binding, soft-Viterbi tag handling, and
   the PN511/PN63 LFSR-vs-spec mismatch.
3. **Watchdogs at multiple levels are how you survive marginal
   signals.** Recovering from a brief gap is more achievable than
   eliminating gaps entirely.
